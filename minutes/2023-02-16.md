### Element Capture
Elad Alon

EA: Sample use case, two apps in same tab.  Docs on the left side, meet on the right side.  
‚Ä¶Applications are cross-origin.  
‚Ä¶Participants may join from other applications.  Want to present back to them, without videos.  
‚Ä¶Limit capture to a region on the page.  Take an element on the page, use it as input to crop capture to that element.  
‚Ä¶Drawbacks for region capture are occluded and occluding content.  
‚Ä¶Could be sharing, there is a drop-down list that appears in the capture.  
‚Ä¶Private data (e.g. messages) could appear in the captured region.  
‚Ä¶Instead, crop based on an element and the descendants of that element in the DOM.  
‚Ä¶With element level capture, you get that element descendants but nothing else.  
‚Ä¶Multiple problems with element.captureStream().  
‚Ä¶Cross-origin content is unsupported.  
‚Ä¶Supporting it would require cross-origin isolation, opt-in by embedded content, and user prompt  
‚Ä¶Proposal instead: Add an option to exclude occlusions with region capture.  
‚Ä¶Re-uses prompt for region capture, does not require cross-origin isolation.  
‚Ä¶Can crop even captures of other tabs.  
‚Ä¶User is giving permission to capture the entire tab.  
‚Ä¶Audio is out of scope for now.  
HA: Some security implications, if content is offscreen or invisible.  
‚Ä¶Perhaps yes, if the element is video, you don‚Äôt want the video to stop.  
EA: When the user is consenting to self-capture, they are consenting to capturing content that is not visible to the user (low opacity, piecemeal, etc.)  
ABK: Useful for dialpad.  How will we access third-party applications (e.g. Slides)?  No API or mechanism to access them.  
EA: If there is a crop target, you can make a business decision to set up an API between them.  
MF: Could the user agent consume crop targets to offer the users?  
EA: Only useful in cases where it‚Äôs not necessary.  
HF: Capture the crop target.  Is there the possibility to exclude part of the tree?  
EA: Better as a later extension?

### Agent Screen Capture
Arnaud Budkiewicz

ABK: Dialpad provides AI-powered customer call center   
‚Ä¶With Agent Screen Capture contact center managers record agent screens during calls.  
‚Ä¶Provide coaching and feedback to agents.  
‚Ä¶Recording is taken during call and is saved to Dialpad for access to managers.  
‚Ä¶Only one API is not standard, desktopCapturer.getSources  
‚Ä¶No Web API to capture all screens without user action  
‚Ä¶MediaRecorder does not support AV1, HW acceleration has a negative impact  
‚Ä¶Pressure on upstream link (1.5 MBps/second/screen) during upload.  Looking at RTCDataChannel  
RZ: What are the limitations?  
ABK: No API to capture all the screens, and skip the picker.  
RZ: The use case is monitoring?   
ABK: One of the use cases, we capture all the screens. There is nothing for the user to select.  
RZ: Some privacy/security concerns handled by this team.  
MH: Why is hardware encoding a negative?  
ABK: The [bitstream size] is too large. This is why AV1 support in MediaRecorder would be nice to have.  
NO: The user agent could allow this in the enterprise context so that there is no user action needed.
